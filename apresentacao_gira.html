Sessão: JWT e Cookie HttpOnly

Arquivo: controllers/authController.js

// 1. Assinatura do Token
const token = jwt.sign(
  { id: user.id, role: user.role, name: user.name },
  process.env.JWT_SECRET,
  { expiresIn: '1h' }
);

// 2. Envio via Cookie com blindagem
res.cookie('jwt', token, {
  httpOnly: true,                                 // 👈 Impede roubo via JavaScript
  secure: process.env.NODE_ENV === 'production',  // Requer HTTPS
  sameSite: 'lax',
  maxAge: 1 * 60 * 60 * 1000
});

Token JWT (jwt.sign)

O Payload contém apenas dados seguros (id, role, nome). Expira rigorosamente em 1 hora.

Cookie de Segurança

Utilizamos atributos httpOnly (protege do JavaScript) e sameSite=lax (protege de Cross-Site).

Limitação Local: O atributo secure (apenas HTTPS) só é ativado em produção. Como o ambiente local roda HTTP, o cookie secure fica desabilitado intencionalmente.

Revogação: Logout e Blacklist

export const logout = async (req, res) => {
  // 1. Registra a saída na trilha de auditoria
  await logAudit(req.user.id, 'logout', 'Usuário efetuou logout.', req.ip);
  
  // 2. Invalida o token inserindo na blacklist
  const token = req.cookies.jwt;
  if (token) {
    await pool.query('INSERT INTO token_blacklist (token) VALUES (?)', [token]);
  }

  // 3. Remove o cookie do navegador
  res.clearCookie('jwt');
  res.json({ message: 'Logout efetuado com sucesso.' });
};

RBAC (Role-Based Access Control)

Arquivo principal: middlewares/rbacMiddleware.js (função requireRole()) 

export const requireRole = (roles) => {
  return (req, res, next) => {
    if (!req.user || !roles.includes(req.user.role)) {
      return res.status(403).json({ message: 'Acesso negado. Perfil sem permissão.' });
    }
    next();
  };
};
  • Acesso interceptado: Caso o perfil não tenha permissão, a API retorna 403 Forbidden.
  • Onde é utilizado:
    • routes/users.js
    • routes/tickets.js
    • routes/audit.js
Demonstração Prática (Isolamento via API): O Postman prova que a validação não está apenas no frontend (React).

Trilha de Auditoria e Limitações

Arquivo: middlewares/auditMiddleware.js (função logAudit()) 

export const logAudit = async (userId, action, details, ipAddress) => {
  try {
    await pool.query(
      'INSERT INTO audit_logs (user_id, action, details, ip_address) VALUES (?, ?, ?, ?)',
      [userId, action, details, ipAddress]
    );
  } catch (error) {
    console.error('Falha ao registrar auditoria:', error);
  }
};
  • Informações Registradas: usuário (id), ação, detalhes (ex: email alterado), IP e Data.
  • Eventos Suportados: login, logout, create_user, update_user, delete_user, change_status, delete_ticket.
  • Transparência no Log:
    • Senhas nunca são registradas no log (apenas ID e e-mail).
    • Limitação Conhecida: Falhas de login (senha incorreta) não estão sendo auditadas no banco.

Camadas Adicionais de Defesa

Arquivo: server.js

// Limpeza XSS (Cross-Site Scripting)
app.use(xss());

// Helmet (Proteção de cabeçalhos)
app.use(helmet());

// Rate Limit para Login (Mitiga Força Bruta)
const loginLimiter = rateLimit({
  windowMs: 15 * 60 * 1000, // 15 minutos
  max: 5,                   // bloqueia no 5º erro
  message: 'Muitas tentativas. Bloqueado.'
});
app.use('/api/auth/login', loginLimiter);
  • Proteções Ativas:
    • Helmet: remove headers sensíveis e protege contra sniff.
    • xss(): limpa inputs maliciosos protegendo contra Cross-Site Scripting.
    • CORS: restringe o consumo apenas à origem autorizada.
  • Rate Limit:
    • Geral da API: 100 requisições a cada 15 minutos.
    • Rota de Login: Apenas 5 tentativas por IP a cada 15 minutos.

Vulnerabilidades

Ameaças e Resposta a Incidentes

Principal Vulnerabilidade: JWT_SECRET Fraco

O segredo usado permite falsificação de tokens. Um atacante não precisa roubar senhas do banco para se tornar administrador.

.env.example (Recomendado)

JWT_SECRET=super_secret_key_change_me

.env local (Atual) 

JWT_SECRET=123456

Fluxo de Resposta a Incidentes

  • 1. Detectar: Investigar audit_logs.
  • 2. Conter: Adicionar os tokens suspeitos na token_blacklist.
  • 3. Preservar: NÃO apagar ou alterar os registros do banco (Evidência Pericial).
  • 4. Erradicar e Recuperar: Alterar o JWT_SECRET imediatamente.
